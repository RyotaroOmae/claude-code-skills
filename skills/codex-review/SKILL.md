# /codex-review — Codex によるコードレビュー

MCP 経由で接続されている Codex (`mcp__codex__codex`) を呼び出し、
コードレビューを依頼するための skill。観点は **汎用** で、バグ・設計・
可読性をまとめて見させる。

Arguments passed: `$ARGUMENTS`

---

## レビュー対象の決定

`$ARGUMENTS` をパースする。

### 引数なし → カレント git リポジトリの差分

1. `pwd` で cwd を確認し、`git rev-parse --is-inside-work-tree` で
   git 管理下か確認する。git 管理下でなければ「対象を引数で指定してください」
   と返して終了。
2. 以下を順に試し、最初にひっかかったものを差分として採用する：
   - `git status --short` に変更ありなら、未コミット変更 (staged + unstaged) を
     対象。`git diff HEAD` で全差分を取得。
   - 未コミット変更がなく、現ブランチが `main`/`master` 以外なら、
     `git merge-base HEAD origin/main` (失敗したら `main`、`master` 順で fallback)
     からの差分を対象。`git diff <base>...HEAD`。
   - どちらもなければ「レビュー対象がありません」と返して終了。
3. `wc -l` などで差分サイズを確認。10,000 行を超える巨大差分は
   「`<引数>` で対象を絞ってください」とユーザーに確認する。

### 引数あり

- 引数がファイル/ディレクトリパスっぽければ、それを対象にする。
  存在確認後、その内容を Codex に直接読ませる (`cwd` を渡せば Codex 側で読める)。
- パスでなく自然言語の指示なら、追加のレビュー観点として Codex への
  プロンプトに混ぜる。

---

## Codex 呼び出し

`mcp__codex__codex` を以下の設定で呼ぶ：

- **`cwd`**: 現在の作業ディレクトリ (cwd を絶対パスで渡す)
- **`sandbox`**: **必ず `"danger-full-access"`** を指定する。
  このサーバ (CentOS 7 / kernel 3.10) では Landlock も bubblewrap も
  利用できず、`read-only` / `workspace-write` は sandbox 初期化に失敗して
  Codex 呼び出しがエラーで落ちる。レビューは元々書き込みしないので
  実害なし。`read-only` を試して fallback、ではなく**最初から
  `danger-full-access` を指定**する。
- **`approval-policy`**: `"never"` (非対話で完結させる)
- **`model`**: 指定しない (MCP 側のデフォルトに任せる)
- **`config`**: `{"model_reasoning_effort": "high"}` を指定する。
- **`prompt`**: 下記テンプレート

### プロンプトテンプレート

```
あなたは熟練のコードレビュアーです。以下のコード/差分を汎用的な観点で
レビューしてください。

# 観点
- **正確性 / バグ**: ロジックの誤り、エッジケース漏れ、競合状態、null 安全性、
  リソースリーク、例外処理の抜け、型の不整合
- **設計 / アーキテクチャ**: 責務分割、結合度、抽象の妥当性、API 設計、
  既存コードとの一貫性、よりシンプルな書き方の有無
- **可読性 / 保守性**: 命名、関数長、コメントの過不足、マジックナンバー、
  デッドコード、テスト可能性

# 出力フォーマット
重要度順にまとめる。各指摘は以下の形式：

- **[重要度: high|medium|low] <ファイル>:<行> — <一言で問題>**
  - 何が起きるか / なぜ問題か (1-2 文)
  - 推奨される修正方針 (具体的に。可能ならコード例)

最後に「総評」として 2-3 文で全体の所感をまとめてください。
問題が無ければ "no issues found" と明記してください。
冗長な前置きや要約は不要です。指摘そのものだけを書いてください。

# レビュー対象
<ここに差分 / ファイルパス / 追加観点を埋め込む>
```

差分をプロンプトに埋め込むとき：

- パスベースで Codex に読ませる場合: 「以下のファイルを読んでレビューしてください: `path/to/file.py`」のように指示。
- diff をそのまま貼る場合: 「以下の git diff をレビューしてください:」の後に diff を貼る。
- ユーザーが追加観点を引数で渡した場合は「特に以下の観点を重点的に見てください: `<追加観点>`」を末尾に付ける。

---

## 結果の扱い

1. Codex の出力を**そのまま**ユーザーに見せる (要約しない)。Codex のレビュー
   そのものに価値があるため、上書きや勝手な省略はしない。
2. Codex が `threadId` を返してきたら、ユーザーが追加質問できるよう
   「`mcp__codex__codex-reply` で続きの会話が可能です (threadId: `<id>`)」と
   一言添える。
3. Codex 呼び出しが失敗 (MCP 未接続など) したら、エラー内容と
   `claude mcp list` で接続状態を確認するよう案内する。

---

## 実装メモ

- **二重作業を避ける**: 自分でレビューせず、Codex の出力を尊重する。
  ユーザーが「Claude 側の意見も欲しい」と言ったときだけ自分の所感を追加する。
- **機微情報の注意**: sandbox は `danger-full-access` で動かしているため
  Codex 側に書き込み・ネット制限がない。ユーザーが明示的に渡さない限り、
  こちらから `.env` や鍵ファイルを Codex に読ませない。
- **Codex 失敗時に自分でレビューしない**: MCP 呼び出しが落ちた場合は
  原因 (sandbox 指定ミス / MCP 未接続など) をユーザーに報告して止める。
  Claude 側で勝手にレビューを代行しない (ユーザーが明示要求した時のみ)。
- **巨大差分**: 10k 行超なら 1 回で投げず、ファイル単位 or ディレクトリ単位に
  分割してユーザーに確認してから複数回呼ぶ。
