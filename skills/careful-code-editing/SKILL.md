# careful-code-editing — 慎重なコード編集の作法

コードを書く・直す・レビューする際に、AIが陥りやすい失敗
（早合点・過剰設計・余計な周辺変更・検証漏れ）を減らすための行動ルール。

このスキルが参照されたら、以下のルールを **そのタスクの間** 守る。
ただし末尾の「軽微な変更の例外」を最優先で読み、簡単な作業まで止めないこと。

Arguments passed: `$ARGUMENTS`（対象や注意点があれば加味する）

---

## 1. 使う場面

- 新規実装
- バグ修正
- リファクタリング
- レビュー指摘への対応
- テスト追加
- 既存コードの編集全般

## 2. 着手前に確認すること

- **曖昧さは言語化する。** 要求に解釈の余地があれば、採用する前提を一文で明示してから進む。
- **解釈が割れるなら勝手に1つに決めない。** 影響が大きい分岐は、選択肢を短く並べて確認する。
- **より小さい解決策が見えたら、書く前に提案する。** 大改修の前に「最小案で足りないか」を一度問う。
- 既存コードを読む前に「直す」と決めない。まず関連箇所を読み、現状の意図を把握する。

## 3. 最小実装

- 要求されていない機能・オプション・設定を足さない。
- 1か所でしか使わない処理を、再利用を見越して抽象化しない。
- 「将来必要かも」だけを根拠に、層・引数・分岐を増やさない。
- 書き終えたら一度見直し、自分が追加・変更した差分内で、より短く・明確にできる箇所を縮める（既存コードには手を入れない）。
- 推測でロジックを足すより、不明点を1つ質問する方が速い場面を選ぶ。

## 4. 既存コードを編集するとき

- 依頼範囲外のコード・コメント・空白・整形を変えない（差分を最小に保つ）。
- 既存の設計・命名規則・コードスタイルに合わせる。自分の好みに寄せない。
- ついでのリファクタリングをしない。気づいた改善点は報告に回す。
- 既存のデッドコードを見つけても削除せず、報告に留める。
- 整理してよいのは「自分の変更で不要になった」import・変数・関数だけ。

## 5. 検証

- バグ修正では、可能なら **失敗を再現するテストを先に書き**、修正で緑にする。
- 実装タスクでは、着手時に「完了の条件」を一文で決めておく。
- 変更内容から、走らせるべき test / lint / 型チェック / build を判断して実行する。
- 実行できない検証があれば、その理由と「未検証のため残るリスク」を明記する。
- 「動くはず」で終わらせない。確認したことと、していないことを分けて述べる。

## 6. 報告のしかた

- 非自明な作業は、最初に短い計画（数行）を出してから着手する。
- 完了時は次の3点を簡潔にまとめる：**変更内容 / 検証結果 / 未確認事項**。
- 差分が想定より大きくなった場合は、なぜ必要だったかを一言添える。

## 軽微な変更の例外（過剰確認の禁止）

次のような明白で小さい作業は、確認や計画提示で止めずに即実行する。

- typo・コメント・ログ文言・1行の定数値の修正
- 明確に指示された1ファイル・数行の編集
- 解釈の余地がなく、影響範囲が局所的な変更

迷ったときの基準：**「間違えても安く戻せるか」**。安く戻せるものは進める。
戻すのが高くつく（広範囲・データ・公開物・不可逆）なら、上のルールに従って確認する。
