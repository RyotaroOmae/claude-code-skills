# explain-code — コードを責任者に「理解させる」

Claude Code が書いた変更、または指定された既存コードを、**責任者であるユーザーが
中身まで理解し、自分の言葉で説明できる**状態にするためのスキル。単なる要約ではなく、
「何を・なぜ・どんなツールで・どう動くか」を解きほぐし、最後にクイズで定着を確認する。

> **モデルについて**: このスキルは frontmatter の `model: sonnet` により Sonnet で動く
> （説明タスクは軽量モデルで十分なため）。この上書きは**このターン限り**で、
> 次のユーザー発話で元のモデルに自動で戻る。手動の復元操作は不要。

このスキルが参照されたら、以下の手順を踏む。

Arguments passed: `$ARGUMENTS`（説明対象のファイル/ディレクトリがあれば、それに絞る）

---

## 1. 使う場面

- Claude Code がコードの実装・修正・リファクタリングを一通り終えたとき
- ユーザーが「いま何を変えたのか／このコードがどう動くのか説明して」と求めたとき
- 引数で指定されたファイル/ディレクトリの中身・意図・仕組みを腹落ちさせたいとき

## 2. 対象範囲の特定

1. **引数があれば**、そのファイル/ディレクトリを対象にする。
2. **引数が無ければ**、直前までの会話で Claude が編集・作成したファイルを対象にする。
   - まず会話履歴から対象候補を整理する。
   - git 管理下なら `git status` / `git diff` で裏取りする。git 非管理の
     作業ディレクトリなら会話の文脈から編集ファイルを特定する
     （会話が長い・圧縮された場合は記憶に頼りきらず、次の3で確認を取る）。
3. 対象が曖昧なときは、推定した対象ファイルの一覧を先に提示して確認を取る。

## 3. 説明の組み立て（チャット出力）

対象ファイルを実際に Read した上で、次の順で簡潔にまとめる。
**推測で書かず、必ずコードを読んでから書く。**

まず、対象が**「変更」なのか「既存コードの理解」なのか**を見極めてモードを選ぶ。
判断基準: 直前の会話で Claude が編集した／git で差分が確認できるなら **変更モード**。
引数で指定された既存ファイルで変更履歴が確認できないなら **理解モード**
（変更前提で「新規/修正」「編集の目的」を推測で書かない）。

**変更モードの場合:**
1. **変更したファイル一覧** — ファイルごとに「新規 / 修正」と一行要旨。
2. **各編集の目的** — そのファイル/関数で何を達成したかったか（なぜ）。
3. **コードの流れ** — 入口から出口までの処理の道筋を、関数・データの流れが
   追えるように記述する。複数ファイルにまたがる場合は呼び出し関係を示す。
4. **責任者が確認すべき点** — 前提・副作用・未検証事項・注意点を箇条書きで。

**理解モードの場合:**
1. **対象ファイル一覧** — ファイルごとに役割の一行要旨（「新規/修正」は付けない）。
2. **目的・役割** — そのファイル/関数が何のために存在し、何をするか。
3. **コードの流れ** — 変更モードと同じく入口→出口の処理の道筋・呼び出し関係。
4. **責任者が確認すべき点** — 前提・副作用・注意点を箇条書きで。

冗長な逐語解説は避け、責任者が判断するのに必要な粒度に留める。

## 4. 依存ツール・ライブラリの解説（このスキルの肝・その1）

「どういうツールで動いているのか」を責任者が把握できるよう、コードが依存する
**外部ライブラリ・モジュール・コマンドの役割**を明示する。名前を挙げるだけでなく、
**この変更の文脈で何をしているか**まで一段掘り下げる。

- **Python**: `import` 文を洗い出し、主要なものを役割で分類して説明する。
  - 例: `numpy`（数値配列・ベクトル演算）, `xarray`（NetCDF/ラベル付き多次元配列）,
    `netCDF4`（NetCDF I/O 低レベル）, `matplotlib`（作図）, `pandas`（表形式）,
    `argparse`（CLI 引数）, `pathlib`（パス操作）。
  - そのライブラリ**特有の作法**（例: `xarray` の `.sel()` によるラベル選択、
    `numpy` のブロードキャスト、`matplotlib` の Figure/Axes 階層）が
    このコードのどこで効いているかを示す。
  - バージョン依存・環境依存が疑わしいときに限り `pip show <pkg>` で確認してよい。
- **Fortran**: `use <module>` と `call <subroutine>` を洗い出し、どのモジュール由来か、
  外部ライブラリ（MPI・NetCDF・SCALE 内部モジュール等）かを区別する。
  - 例: `use mpi`（並列）, `use netcdf`（I/O）, SCALE の `scale_*` モジュール群。
  - `intent(in/out)`、`allocatable`、配列の次元・インデックス規約（i,j,k の順序）など、
    **Fortran 特有でバグの温床になりやすい点**にも触れる。
- 言語共通: ファイル I/O・並列・乱数・時間計測など「裏で何が動いているか」を、
  責任者がブラックボックスのまま放置しないよう一言で言語化する。

**根拠はコードと実在するライブラリの挙動のみ。** 読み取れない依存は「要確認」とする。

## 5. 最小コードへの蒸留（このスキルの肝・その2）

責任者が**本質を一目で掴める**よう、コードから「本質でない防御的処理」を削ぎ落とした
**最小版**を提示し、元コードと対比して解説する。

- 削ぎ落とす対象（＝本質ではない、どうでもよい部分）の例:
  - ファイル/ディレクトリ存在チェックや「無い場合の分岐」
  - try/except・エラーハンドリング・ログ出力・進捗表示
  - 引数パース・デフォルト値の埋め合わせ・型変換の保険
  - キャッシュ・早期 return・境界条件の場合分けのうち、主筋に関係しないもの
- **残す対象**＝アルゴリズムの主筋・データ変換の核・物理/数式の実体・出力の生成。
- 提示の形:
  1. 「本質だけを残すとこうなる」という**短い擬似/実コード片**（10〜20行目安）。
  2. 「削った部分とその理由」を箇条書き（何を・なぜ本質でないと判断したか）。
  3. 一文で「結局このコードは○○を××している」と要約する。
- 注意: これは**説明用の蒸留であり、実ファイルは書き換えない**。
  最小版はチャット／ドキュメント上にだけ示す。元コードのロジックには一切触れない。

## 6. コメント不足の補完（提案 → 許可制で追記）

説明の過程で、**コメントが無いと意図が追えない箇所**を見つけたら、
解説コメントの**追記候補を提示**する。これは説明スキルなので、
**ユーザーが明示していない実ファイル変更を勝手に行わない**。

- まず「どのファイルのどこに・どんなコメントを足すか」の候補を一覧で示す。
- **実際の追記は、ユーザーが許可したときだけ行う**（「追記して」等の同意後）。
  ユーザーが `/explain-code` を明示呼び出しし、かつ追記に同意した場合に限る。
- 追記してよいのは **コメントのみ**。**コードのロジック・挙動は一切変えない**。
- 対象は、非自明な処理・マジックナンバー・前提条件・なぜその実装かが
  読み取れない箇所。自明な行に冗長なコメントを足さない。
- 既存のコメントスタイル・言語（日本語/英語）・密度に合わせる。
- コメントの内容は、必ず実際のコードの挙動に基づく。推測で書かない。
  挙動が読み切れない箇所は、コメントを足さず説明側で「要確認」として挙げる。
- 追記したら、どのファイルのどこに何のコメントを足したかを報告に含める。

## 7. 理解度クイズ（このスキルの肝・その3）

説明の最後に、責任者の理解が定着したかを確認する**クイズを3〜5問**出す。

- 出題方針:
  - 第4節（ツール/ライブラリ）・第5節（本質）・第3節（流れ）から最低1問ずつ。
  - 「このライブラリは何のために使われている？」「この分岐を消しても本質は変わらない？
    変わる？」「入力 X を与えると出力はどうなる？」など、**理解していないと答えられない**問い。
  - 暗記ではなく、仕組み・因果・トレードオフを問う。引っかけは避け、本質を突く。
- 形式: 番号付きで問いだけを先に提示し、**ユーザーの解答を待つ**。
  解答が返ってきたら採点し、間違い・曖昧な点を補足解説する。
  （ユーザーが「答えも一緒に出して」と望めば、各問の直後に模範解答を付けてよい。）
- クイズはあくまで理解支援。詰問にならないよう、外しても丁寧に補う。

## 8. ドキュメント生成（任意・ユーザーが望んだときのみ）

このスキルの主目的はチャット上での理解支援（第3〜7節）であり、
ファイル生成は**副作用**なのでデフォルトでは行わない。
**ユーザーがドキュメント化を望んだとき**（または「ドキュメントも残して」等の
指示があったとき）に限り、変更解説を Markdown ファイルとして残す。

- 出力先: 作業ディレクトリ直下に `CHANGES_<YYYY-MM-DD>.md`。
  - 同日の既存ファイルが**無ければ**新規作成。
  - **あれば**、その日のファイルの末尾に区切り（`---` と見出し）を付けて追記する。
- 内容: 第3節の項目（ファイル一覧 / 目的 / コードの流れ / 確認点）に、
  第4節（依存ツール・ライブラリ）・第5節（最小コードへの蒸留）・
  提示／追記したコメント一覧を加えたもの。
  - **クイズは問題のみを記載し、模範解答は書かない**（回答前にネタバレさせない）。
    ユーザーの採点が済んだ後で「模範解答も追記して」と望まれたら追記する。
- 後から責任者が単独でレビュー・復習できるよう、ファイル名と該当箇所を明記する。
- `/tmp` 配下には作らない（プロジェクト内に置く）。

## 9. 報告

報告は次の順で行う。

1. チャットに第3〜5節のまとめを出す。
2. 第6節でコメント追記候補を提示した場合は、その一覧と追記の有無（許可済みか）を示す。
3. 最後に第7節のクイズで締め、**ユーザーの解答を待つ**。
4. ドキュメントを生成した場合のみ、「`CHANGES_<日付>.md` に出力した」旨を添える
   （生成していなければ触れない）。

---

## 注意

- これは**理解支援（説明・依存解説・最小化・クイズ）**のスキル。
  コードの**機能変更はしない**。ロジック修正が要ると気づいたら、勝手に直さず報告で指摘する。
- **実ファイルへの書き込みは原則しない**。コメント追記（第6節）はユーザーの同意時のみ、
  ドキュメント生成（第8節）はユーザーが望んだときのみ。
- 説明・依存解説・最小版は実コードに忠実に。読み取れない部分を「たぶんこう」で埋めない。
- 最小コードは**説明用の蒸留**であり、実ファイルには反映しない。
- 秘密情報・環境変数を説明文やドキュメントに転記しない（CLAUDE.md 禁止事項）。
- `model: sonnet` の上書きはこのターン限り。次の発話で元モデルに自動復帰する（手動操作不要）。
