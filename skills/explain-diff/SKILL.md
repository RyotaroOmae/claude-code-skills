# explain-diff — 既知との「差分」だけで理解させる

人は何かを理解するとき、ゼロから全部を覚えるより **「既に理解していること」との
差分(デルタ)だけ**を押さえる方が効率的で、これは認知科学で **学習の転移
(transfer of learning)** と呼ばれる。このスキルはその発想をそのまま実装する。

**既知の主アンカー(基準)を一つ定め、共有部分は意図的に省き、差分だけを既知の言葉に
接続して解説する。** 最後に差分クイズで定着を確認する。

> **explain-code との違い**: `explain-code` は対象を**ゼロから自己完結で全部**
> 解説する（ファイル一覧・最小蒸留・ドキュメント生成まで）。本スキルは逆に
> **共有部分を捨て、差分だけ**を解説する。「○○との違いだけ知りたい」「△△は
> 分かってる前提で」なら explain-diff、「全部腹落ちさせたい」なら explain-code。

> **モデルについて**: frontmatter の `model: sonnet` により Sonnet で動く
> （説明タスクは軽量モデルで十分なため）。この上書きは**このターン限り**で、
> 次のユーザー発話で元のモデルに自動で戻る。手動の復元操作は不要。

このスキルが参照されたら、以下の手順を踏む。

Arguments passed: `$ARGUMENTS`（第1要素＝説明対象、第2要素以降＝基準/アンカーがあれば）

---

## 1. 使う場面

- 「○○と同じだけど**ここだけ違う**」コードを理解したいとき。
- 既存スクリプトから**派生**させたスクリプト（元との差分が知りたい）。
- おなじみのライブラリ・パターンの**新しい使い方**（既知の作法からの差分）。
- 「△△は分かってる前提で、それとの違いだけ説明して」と求められたとき。

ゼロから全部理解したい・派生元が存在しない単独の新規コードなら、本スキルではなく
`explain-code` の方が適する。

## 2. 対象とアンカーの確定（入口・肝その1）

**差分は「何を基準にするか」で意味が変わる。** ここを曖昧にしたまま進めない。

### 2-1. 対象（差分を理解したいコード）
1. **引数の第1要素があれば**そのファイル/ディレクトリを対象にする。
2. **無ければ**直前の会話で Claude が編集・作成したコードを対象にする
   （git 管理下なら `git status` / `git diff` で裏取り。非管理なら会話文脈から特定）。
3. 対象が曖昧なら、推定した対象を先に提示して確認を取る。

### 2-2. アンカー（既知の基準）
**主アンカーは必ず1つに定める。** 補助アンカー（後述）は必要時のみ。
アンカーの確定経路は2つ:

- **(A) ユーザーが明示している場合** — 引数の第2要素や「○○との差分で」「△△は
  分かってる前提で」等の指示があれば、それを主アンカーとしてそのまま採用する
  （この場合は確認不要）。
- **(B) 明示が無い場合** — Claude が候補を推測し、**必ず提示してユーザーの確認を
  待ってから先へ進む**（推測のまま差分解説を始めない）。推測源は次の3種:
  - (a) **似た既存ファイル** — `Grep`/`Glob` で同種を探す（例: `plot_*.py`、
    同じディレクトリの兄弟スクリプト、関数名が近いもの）。
  - (b) **定番ライブラリ・パターン** — pandas/numpy の標準的な使い方、典型的な
    ループ/IO 構造など。ただしユーザーの既知と決めつけず、「これは既知という
    前提でよいか」を**候補として確認する**（勝手に省略の根拠にしない）。
  - (c) **直前に説明・編集したコード** — この会話で既に理解済みと確認できるもの。

補助アンカー: 主アンカーだけで説明しきれないとき（例: ファイルは A 由来だが
新しいライブラリ作法は B が基準）に限り、補助アンカーを**明示的に**足してよい。
その場合、各差分が**どのアンカーに対する差分か**を出力で必ず示す。

**有効なアンカーを特定できないとき**（似た既存ファイルも、既知と確認できる基準も
無い＝事実上ゼロから理解する単独コード）は、**差分解説を始めない**。
アンカーの指定を依頼するか、「全体をゼロから理解するなら `explain-code` が適切」と
案内して止まる。**無理に定番パターンをアンカーに仕立てない。**

## 3. 共有部分の同定（意図的に省く宣言・肝その2）

アンカーと対象で**同じ・既知**な部分を箇条書きで挙げ、
「**ここは既知として解説を省略する**」と**明示的に宣言**する。
これが転移の核心であり、このスキルの存在意義。

- ユーザーがその場で「そこは分かってる／実はそこは知らない」と正せるようにする。
- もしユーザーが「そこは実は知らない」と言えば、**その部分だけ**を掘り下げる
  （全体をゼロから解説したいなら explain-code に切り替える旨を一言案内してよい）。

## 4. デルタの抽出と解説（肝その3）

対象とアンカーを実際に Read（コードなら `git diff` や対比読み）した上で、
**推測で書かず、必ず根拠を取ってから**書く。

- 差分を **追加 / 変更 / 削除** で構造化する。
- 各デルタを次の形で解説する:
  - 「**アンカーでは X、ここでは Y。なぜ Y にしたか／Y で挙動・出力がどう変わるか**」。
- デルタの**波及**を一段示す — データの流れ・出力・（解析コードなら）物理量や
  単位・次元がどう変わるか。
- **依存ライブラリ**は差分に関わる部分だけ役割解説する。共有しているライブラリの
  一般的な使い方は**再解説しない**（例: 新たに増えた `import` や、同じライブラリの
  新しい関数呼び出しだけを扱う）。
  - Python 例: 追加された `xarray`/`netCDF4`/`matplotlib` の使い方、その差分箇所での
    特有の作法（`.sel()` のラベル選択、Figure/Axes 階層 等）。
  - Fortran 例: 増減した `use`/`call`、`intent`・配列次元・i,j,k の順序など差分の要点。

読み取れない差分・依存は「**要確認**」と明示する。「たぶんこう」で埋めない。

## 5. アンカーへの接続（転移の成立・肝その4）

差分を**既知の言葉に接続**して、転移を成立させる。

- 新概念を既知の言葉で**言い換える**
  （例:「`xr.sel()` は pandas の `.loc` のラベル版を多次元に広げたもの」）。
- 「**既知のメンタルモデルをどう拡張すれば対象を理解できるか**」を一文でまとめる
  （例:「`plot_mynn.py` の理解に『鉛直プロファイルの軸を1本追加する』だけ足せば
  このスクリプトになる」）。

## 6. 差分クイズ（定着確認・肝その5）

最後に、差分の理解が定着したかを確認する**クイズを3〜5問**出す。

- 出題方針 — **差分を理解していないと答えられない**問いにする:
  - 「アンカーと比べてここが違うのは**なぜ**か」
  - 「この差分を消すと挙動は**アンカーに戻るか／戻らないか**」
  - 「アンカーの知識を**この対象に転用**するとどう動くか」
- 暗記ではなく、差分の因果・トレードオフ・転移可能性を問う。引っかけは避ける。
- 形式: 番号付きで問いだけを先に提示し、**ユーザーの解答を待つ**。
  解答が返ってきたら採点し、間違い・曖昧な点を補足する。
  （「答えも一緒に」と望まれたら各問直後に模範解答を付けてよい。）
- クイズはあくまで理解支援。外しても丁寧に補う。

## 7. 報告

報告は次の順でチャットに出す。

1. **対象とアンカー** — 何を、何との差分で説明するか（推測アンカーなら確認込み）。
2. **共有部分（省略宣言）** — 既知として省く部分の箇条書き。
3. **デルタ解説** — 追加/変更/削除と「なぜ・どう変わるか」（第4節）。
4. **アンカーへの接続** — 既知の言葉への言い換え・メンタルモデルの拡張（第5節）。
5. **差分クイズ** — 第6節で締め、**ユーザーの解答を待つ**。

---

## 注意

- これは**差分による理解支援**のスキル。**共有部分・既知部分を冗長に再解説しない**
  （それをするなら explain-code）。
- **実ファイルへの書き込みはしない**（allowed-tools に Edit/Write を含めない）。
  コメント追記・ドキュメント生成が要るなら explain-code 側の責務。
- 差分・アンカーが読み取れない箇所は「要確認」とする。推測で説明を埋めない。
- コードの**機能変更はしない**。ロジック修正が要ると気づいたら、勝手に直さず報告で指摘する。
- 秘密情報・環境変数を説明文に転記しない（CLAUDE.md 禁止事項）。
- `model: sonnet` の上書きはこのターン限り。次の発話で元モデルに自動復帰する（手動操作不要）。
